# OpenClaw 公共新闻库技能

## 必须遵守的安全准则

每次运行本skill，均需认真阅读文档内的限制说明！首先，你必须遵守如下的安全限制：

- 当无法接收到发布成功，服务器正常运行的反馈时，不允许再次发送或是私自发送测试请求，立刻停止运行skill，并直接向用户报错，附带网站输出的报错信息。
- 在上一条的基础上，只有在询问用户是否能发送测试请求并得到用户明确的肯定回答后，才能发送测试信息。注意！你仅仅只能发送有限的，不超过三条的测试信息，若需要发送更多的测试信息，需要再次询问获得用户的明确同意。
- 在上两条的基础上，你没有任何权限自主对用户已有的项目文件进行修改与添加。如非必要不选择自主修复bug。只有当用户明确表达帮忙修复问题时才能介入。

除了如上的安全限制，下文也有许多需要你注意的限制。这些限制同样不可违反！绝对不能违反这些限制，每次运行都需要检查是否违反这些限制，随时检查是否违反。如果发现违反，立刻停止服务并向用户说明情况！待用户回复之后才能重新开始。


## 何时使用

当用户提出以下需求时使用本技能：

- 建立或维护公共新闻库（关键词、概述、原文链接）。
- 向新闻库写入新闻数据，或按关键词检索新闻。
- 将新闻动态页面与新闻库联动展示。
- 做新闻库数据质量治理（去重、补全、来源校验）。

不适用于仅做价格采样和价格趋势图分析的场景。

## 数据模型约定

新闻库核心字段：

- `keyword`：关键词（如“比特币”“黄金”）
- `summary`：新闻概述（建议 1-3 句）
- `source_url`：原文链接（可直达）
- `title`：新闻标题（可选）
- `source_name`：来源名称（可选）
- `published_at`：原文发布时间（可选）

## 接口清单

### 写入新闻库（鉴权）

- `POST /api/v1/openclaw/news/library`
- 需要请求头：`X-Api-Key`

请求体示例：

```json
{
  "keyword": "比特币",
  "summary": "BTC在议息会议前波动扩大，成交量提升。",
  "source_url": "https://example.com/btc-news-1",
  "title": "比特币波动放大",
  "source_name": "Example News",
  "published_at": "2026-04-08T12:00:00+08:00"
}
```

### 查询新闻库（公开）

- `GET /api/v1/public/news/library?limit=100&keyword=比特币`
- 可按关键词过滤，默认返回最近数据

## 工作流程

按以下顺序执行：

```text
News Library Checklist
- [ ] 确认 OPENCLAW_NEWS_DATABASE_URL 已配置
- [ ] 写入 1 条测试新闻
- [ ] 查询并确认可返回数据
- [ ] 在新闻动态页面验证展示
- [ ] 若有重复，执行去重策略
```

## 常用命令模板

### 1) 写入

```bash
curl -sS -X POST "$BASE_URL/api/v1/openclaw/news/library" \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: $API_KEY" \
  -d '{
    "keyword":"比特币",
    "summary":"BTC在议息会议前波动扩大，成交量提升。",
    "source_url":"https://example.com/btc-news-1",
    "title":"比特币波动放大",
    "source_name":"Example News",
    "published_at":"2026-04-08T12:00:00+08:00"
  }'
```

### 2) 查询

```bash
curl -sS "$BASE_URL/api/v1/public/news/library?keyword=%E6%AF%94%E7%89%B9%E5%B8%81&limit=20"
```

## 新闻动态页面联动约定

- 页面左栏展示新闻库列表（可关键词筛选）。
- 页面右栏展示详情（标题、概述、原文链接）。
- 原文链接必须可点击直达，打开新标签页。
- 页面支持勾选批量删除，删除后应立即刷新列表并清空已选状态。

## 与价格分析技能联动（关键新增）

当用户希望“抓到新新闻后快速给出价格判断”时，本技能负责提供触发输入，交由 `openclaw-price-analysis-reporting` 执行联合分析。价格时间序列默认来自 **OpenClaw 上报** 的 `observations/ingest` 与公开 **`GET /api/v1/public/monitoring/...`** 读库，而非服务端网页抓取（参见主项目 `README.md` 与 `openclaw-news-publisher-enhanced` 的 §8.6）。

### 联动触发条件

- 写入新闻成功后，命中监测关键词（如“比特币”“黄金”“原油”“羽毛球”）。
- 或查询时发现 `news_hours` 窗口内存在新增新闻（相对上次分析时间戳）。

### 触发时输出给分析技能的最小上下文

- `keyword`
- `monitor_id`（若有映射）
- `news_hours`（建议 24/72/168）
- `horizon`（`24h` / `72h` / `7d`）
- 最近 1-5 条高相关新闻（`title`、`summary`、`source_url`、`source_name`、`published_at`/`created_at`）

### 关键词到 monitor 的映射建议

- 优先使用用户提供的映射表（最可靠）。
- 无映射表时，按关键词精确匹配监测任务名；再降级到包含匹配。
- 仍无法匹配时，返回“需补充 monitor 映射”，不做强行触发。

### 事件优先级规则（用于短时间快报）

- `P0`：宏观冲击类（监管、地缘冲突、利率决议、OPEC 政策）且与监测关键词强相关。
- `P1`：行业/供需类（库存、产能、渠道价格变动）。
- `P2`：一般资讯或重复转载。
- 快报默认优先处理 `P0/P1`，`P2` 仅作背景信息。

## 数据质量规则

- 禁止编造 `source_url`。
- `summary` 不可为空，不可仅复制标题。
- 同关键词同链接重复入库时，建议保留最新一条（或做上游去重）。
- `published_at` 缺失时可为空，但应在页面标识“未知发布时间”。
- 触发联合分析时，关键结论至少可追溯到 1 条新闻证据（标题+链接+时间）。
- 同一 `source_url` 在短时间重复抓取时，应视为同事件，避免重复触发快报。

## 异常处理

- `503`：未配置 `OPENCLAW_NEWS_DATABASE_URL`
- `401`：`X-Api-Key` 无效
- `500`：数据库连通或表结构异常

出现错误时需反馈：

- 失败步骤
- 接口与状态码
- 下一条建议排查命令
